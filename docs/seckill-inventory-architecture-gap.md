# 秒杀库存架构差异与 A 方案演进记录

## 总览

当前 `mall-seckill` 已演进为“Redis 资格准入 + MySQL 异步落账”的秒杀链路：入口限流后用 Redis Lua 做库存资格预扣和一人一单，HTTP 链路写入扣减快照与可靠消息，RabbitMQ 异步创建订单并在结果确认阶段落账 MySQL 秒杀库存。`inventory_hotspot_seckill_architecture_markdown/inventory_hotspot_seckill_architecture.md` 描述的是资损安全优先的库存扣减系统：先把库存扣减迁到事务型数据库，再通过单行性能治理和中心桶 + 分桶模型支撑热点流量。

两者最大的差别不是组件数量，而是库存权威源和账本闭环。当前实现更像“快速挡流 + 异步下单”，文档架构更强调“库存事实可追溯、扣减可幂等、失败可回补、热点可水平拆散”。

## 核心差异

### 1. 库存权威源不同

当前 `mall-seckill` 的主路径是 Redis Lua 扣减 `seckill:stock:{activityId}:{skuId}`，Redis 是必需的资格准入账本；Redis 不可用时直接拒绝，不再降级到 MySQL 条件扣减。MySQL 的 `seckill_sku.stock` 在秒杀结果确认时异步扣减，配合 `seckill_stock_snapshot` 保留可追溯账本。

### 2. 双库存扣减已收敛

秒杀期间以 `seckill_sku.stock` 作为成交库存账本，订单服务消费秒杀下单消息后不再调用 `mall-product` 扣商品库存，秒杀订单关单时也不释放商品库存。这样避免了 `seckill_sku.stock` 和 `sku_stock.stock` 双扣/双放导致的口径冲突。

文档架构的做法是统一库存扣减模型：一次扣减必须能通过快照和流水解释清楚，后续回补也必须能找到原始扣减位置。

### 3. 扣减快照已补齐，仍待进一步流水化

文档阶段三把交易快照作为核心能力：扣减时记录业务标识和扣减位置，重试时用它保证幂等，失败或退款时用它做精确回补。当前 `mall-seckill` 已新增 `seckill_stock_snapshot`，记录 `requestId`、活动、SKU、用户、数量和状态；但还没有独立库存流水表，也没有中心桶/分桶扣减位置。

### 4. 当前不是分桶架构

当前表结构是单行库存：`seckill_sku(activity_id, sku_id, stock)`。文档目标是中心桶 + N 个分桶：C 端请求扣分桶库存，中心桶异步汇总总库存，分桶耗尽后通过存活桶列表和调拨机制继续承接流量。

### 5. 当前限流与隔离还比较粗

当前 Sentinel 资源主要是全局 `seckill-submit`。文档阶段二还包含按热点商品识别、系统预热、连接池治理、冷热流量隔离等能力。这些属于吞吐治理，应该在账本正确后再做。

## 推荐演进路线

### A. 面试展示型演进

目标是补齐正确性账本，不追求真实 8W QPS。保持现有技术栈，优先完成：

- 秒杀期间以秒杀库存为成交库存，订单服务不再二次扣商品库存。
- 增加扣减快照，记录 `requestId`、活动、SKU、用户、数量和状态。
- 订单创建失败时，按快照回补秒杀库存。
- 避免秒杀订单关单时错误释放商品库存。

### B. 本地压测型演进

在 A 的基础上继续做动态限流、热点 SKU 资源拆分、预热、压测脚本和指标观测，用本地压测说明瓶颈位置。

### C. 生产热点型演进

引入中心桶、分桶、分片路由、异步总库存汇总、调拨锁和存活桶列表。这需要真实分库/分片或等价基础设施；当前单库项目只能模拟模型，不能证明文档里的 8W QPS 能力。

## 本次实现边界

本次选择 A 方案，聚焦库存账本正确性：

- 新增 `seckill_stock_snapshot`，作为扣减快照表。
- Redis 资格预扣成功后写入 `DEDUCTED` 快照；订单成功后确认快照并异步扣减 MySQL 秒杀库存；订单创建失败后释放快照和 Redis 资格。
- 秒杀订单创建不再调用 `mall-product` 扣商品库存，避免双扣。
- 秒杀订单取消或关单时，不再调用 `mall-product` 释放商品库存，避免未扣先加。
- Redis 启动预热时以 `seckill_sku.stock - DEDUCTED 预约量` 作为可售库存，避免重启后放大未确认资格。

本次不实现中心桶、分桶、动态限流和真实调拨机制。这些应在 A 方案稳定并有压测数据后再进入下一阶段。
