# Seckill Stage 3 Bucket Design

## 背景

`inventory_hotspot_seckill_architecture_markdown/inventory_hotspot_seckill_architecture_md.md` 的阶段三目标是从单库存 6k QPS 演进到 8w QPS 量级。核心手段不是继续优化单行 `UPDATE`，而是引入“中心桶 + N 个分桶”的水平拆分模型：C 端扣减请求打散到多个分桶库存行，中心桶只维护总量、配置和存活桶列表，库存汇总、调拨和碎片整理通过异步或低频链路完成。

当前 `mall-seckill` 已经完成阶段一/二的主要工程基础：

- `seckill_sku.stock/version` 是库存事实源。
- HTTP 提交事务内同步写扣减快照、可靠消息 outbox、单行库存扣减和结果占位。
- RabbitMQ 异步创建秒杀订单，订单结果回传后确认快照或释放库存。
- TairString 只做版本化库存缓存和售罄快速失败，不是库存事实源。
- 热点 SKU 支持显式配置、独立 Sentinel 资源、本地并发保护和启动预热。

当前瓶颈仍然是单库存行。无论入口限流、连接预热、缓存和 outbox 做得多完整，所有成功扣减最终仍排队更新同一条 `seckill_sku` 记录。阶段三要解决的是这个物理热点，而不是再包一层缓存或消息队列。

## 目标

阶段三设计目标分两层：

- 生产语义目标：完整表达文档中的中心桶、分桶、快照幂等、异步总库存、调拨、下线、分片键治理和观测验收。
- 本项目落地目标：先在当前单库或单 OceanBase 环境中实现可演示的分桶模型闭环，证明数据正确性和请求打散逻辑，不宣称达到 8w QPS。

具体目标：

- 把 C 端扣减从单行 `seckill_sku` 改为多分桶库存扣减。
- 快照必须记录本次扣减落在哪个分桶，重试和回补都固定回到原分桶。
- 中心桶不参与 C 端高频扣减，避免变成新热点。
- 总库存展示、售罄判断和监控允许短暂延迟，由异步汇总收敛。
- 分桶耗尽时要能下线，回补或调拨后要能重新上线。
- 低库存不均时通过调拨降低失败率，但调拨不能放大热点。
- 所有设计都要保留当前可靠消息、订单幂等和失败回补闭环。

## 非目标

- 不在当前文档阶段修改业务代码、DDL 或配置。
- 不承诺单机、本地 Docker 或单库环境达到 8w QPS。
- 不把 Redis/Lua 作为阶段三库存事实源。
- 不把 stock-only 或 update-only 压测结果包装成正式秒杀主链路吞吐。
- 不引入跨库分布式事务来实时更新中心桶和分桶。
- 不在首个实现版本做自动热点识别；继续沿用显式热点配置，后续再增强。

## 当前差距

### 库存模型差距

当前 `seckill_sku` 是单行库存模型：

- 一个 `activity_id + sku_id` 对应一条库存记录。
- `stock` 表示可售库存。
- `version` 用于保护 TairString 版本缓存。
- 所有成功扣减都更新同一个 `id`。

阶段三需要：

- 一个中心桶记录总设置库存、总可售口径、总预占口径、分桶配置和存活桶列表。
- 多个分桶记录实际承接 C 端扣减的可售库存。
- 分桶具备状态、分片键、桶编号和版本。
- 同一秒杀 SKU 的分桶可以分散到不同数据库分片或物理实例。

### 快照差距

当前 `seckill_stock_snapshot` 记录 `stock_id`，可以用于失败回补单行库存。阶段三需要把快照扩展为“扣减位置证明”：

- 记录 `bucket_id` 或分桶库存行主键。
- 记录选择桶的策略版本，方便排查路由变化。
- 记录扣减前后的分桶版本或变更流水编号，便于审计。
- 重试时先读快照，不能重新随机选桶。

### 缓存差距

当前 TairString key 是 `seckill:stock-cache:{activityId}:{skuId}`，表示单库存剩余值。阶段三至少需要两个缓存口径：

- 总库存展示缓存：面向 C 端展示和售罄快速失败，允许异步延迟。
- 存活桶列表缓存：面向扣减路由，必须能快速排除已空桶，并在异常时回源数据库。

### 治理差距

当前热点治理能保护入口和数据库并发，但不能突破单行写热点。阶段三需要新增：

- 分桶路由均匀性指标。
- 分桶扣减成功率和失败原因。
- 存活桶数量。
- 异步汇总延迟。
- 调拨次数、调拨冲突、调拨失败。
- 分桶下线和上线事件。

## 方案选择

### 方案 A：单库可演示分桶模型

在当前数据库中新增分桶表和流水表。多个分桶仍在同一个数据库实例内，但每个分桶是不同库存行。该方案能验证正确性、幂等、回补、下线、调拨和异步汇总，不能证明真实 8w QPS。

优点是成本可控，最适合当前项目继续演进。缺点是性能结论只能写“模型闭环”和“相对打散”，不能写生产吞吐。

### 方案 B：生产分片分桶模型

引入真实分片路由，按分片键把不同分桶写到不同逻辑分片或物理实例。该方案最贴近原文，能验证水平扩展能力。

优点是架构真实性最高。缺点是需要多实例数据库、路由规则、压测基础设施和运维脚本，明显超出当前单仓个人项目的默认环境。

### 方案 C：只做多行随机扣减

把一个 SKU 拆成多条库存行，请求随机扣其中一条，不做中心桶、存活桶列表、调拨和异步汇总。

优点是实现最快。缺点是会遗漏阶段三真正复杂的部分，后续补一致性和运维能力会返工。

推荐采用方案 A 作为第一阶段实现边界，并在设计中保留方案 B 的接口位置。这样既不夸大当前项目能力，也不会把阶段三做成残缺的多行随机扣减。

## 数据模型设计

### 分桶库存表

建议新增 `seckill_stock_bucket`，而不是直接把 `seckill_sku` 改造成多角色表。这样能保留当前 SKU 元数据表的清晰职责，也方便灰度阶段同时保留旧单行链路。

字段建议：

| 字段 | 含义 |
| --- | --- |
| `id` | 分桶库存行主键 |
| `activity_id` | 秒杀活动 ID |
| `sku_id` | 商品 SKU ID |
| `bucket_no` | 桶编号，中心桶可用 `0`，业务分桶从 `1` 开始 |
| `bucket_type` | `CENTER` 或 `BUCKET` |
| `shard_key` | 分片键，生产模式用于路由到不同分片 |
| `saleable_quantity` | 可售库存 |
| `occupy_quantity` | 预占库存，主要用于追加、调拨和异步操作中转 |
| `setting_quantity` | 设置总库存，中心桶维护 |
| `status` | `ACTIVE`、`EMPTY`、`DISABLED` |
| `version` | 分桶版本，保护缓存和审计 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

约束建议：

- `UNIQUE(activity_id, sku_id, bucket_no)`。
- `INDEX(activity_id, sku_id, bucket_type, status)`。
- `INDEX(activity_id, sku_id, shard_key)`。

### 分桶配置表

建议新增 `seckill_bucket_config`，记录一个秒杀 SKU 的分桶策略。

字段建议：

| 字段 | 含义 |
| --- | --- |
| `activity_id` | 活动 ID |
| `sku_id` | SKU ID |
| `bucket_count` | 业务分桶数量 |
| `route_mode` | `RANDOM_ACTIVE`、`HASH_USER`、`WEIGHTED_RANDOM` |
| `status` | `ENABLED`、`DISABLED` |
| `strategy_version` | 路由策略版本 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

首版建议使用 `RANDOM_ACTIVE`：从存活桶列表中随机选择一个桶。后续可以增加 `HASH_USER` 降低同一用户重试抖动，但重试幂等仍以快照为准。

### 存活桶列表

存活桶列表可以有两个实现层次：

- 数据库事实源：中心桶或配置表中维护 `survivor_bucket` 字段，存储仍可扣减的桶编号列表。
- 缓存加速层：Redis Set 或 String 存储同一列表，扣减入口优先读缓存，失败时回源数据库。

首版建议事实源放在数据库，缓存作为可失效加速层。缓存异常不能导致库存错误，只能导致回源或快速失败。

### 快照表扩展

当前 `seckill_stock_snapshot` 需要增加分桶相关字段：

| 字段 | 含义 |
| --- | --- |
| `bucket_id` | 实际扣减的分桶库存行主键 |
| `bucket_no` | 实际扣减的桶编号 |
| `strategy_version` | 选择桶时使用的路由策略版本 |
| `change_id` | 对应库存变更流水 ID，可选 |

重试规则：

- 如果 `request_id` 已存在且状态是 `DEDUCTED` 或 `CONFIRMED`，直接返回已有处理语义。
- 如果请求因超时重试，必须根据快照固定扣原分桶，不能重新选择分桶。
- 如果订单失败或取消，需要按快照回补原分桶。

### 库存变更流水表

建议新增 `seckill_stock_change_log`，记录分桶和中心桶的库存变化，供异步汇总和审计使用。

字段建议：

| 字段 | 含义 |
| --- | --- |
| `id` | 流水 ID |
| `request_id` | 业务请求 ID |
| `activity_id` | 活动 ID |
| `sku_id` | SKU ID |
| `bucket_id` | 分桶 ID |
| `bucket_no` | 桶编号 |
| `change_type` | `DEDUCT`、`RELEASE`、`TRANSFER_OUT`、`TRANSFER_IN`、`CENTER_SYNC` |
| `quantity_delta` | 库存变化量，扣减为负，回补为正 |
| `before_quantity` | 变化前数量，可选 |
| `after_quantity` | 变化后数量 |
| `status` | `NEW`、`AGGREGATED`、`FAILED` |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

## 核心流程设计

### 初始化分桶

初始化由后台管理或脚本触发，不在 C 端请求链路内执行。

流程：

1. 创建或更新 `seckill_bucket_config`。
2. 创建中心桶，写入 `setting_quantity`、`saleable_quantity`、`occupy_quantity`。
3. 按 `bucket_count` 创建业务分桶。
4. 把总库存均匀分配到业务分桶，余数按桶编号顺序补齐。
5. 写入存活桶列表。
6. 预热总库存缓存和存活桶列表缓存。

边界：

- 初始化必须幂等，同一 `activity_id + sku_id` 不能重复创建不同配置。
- 分桶数量首版固定，不支持秒杀过程中动态扩桶。
- 如果要增加库存，优先写中心桶 `occupy_quantity`，再异步或低频调拨到分桶。

### 秒杀扣减

扣减流程要保持当前主链路语义：HTTP 线程同步确认库存资格并写 outbox，订单仍异步创建。

流程：

1. 入口执行 Sentinel 和热点并发保护。
2. 加载活动、SKU 元数据和分桶配置。
3. 查询或读取存活桶列表。
4. 如果请求已有快照，使用快照中的 `bucket_id` 固定重试。
5. 如果没有快照，从存活桶列表选择一个桶。
6. 在同一数据库事务内：
   - 检查一人一单活动快照。
   - 插入 `seckill_stock_snapshot(DEDUCTED)`，记录桶信息。
   - 写 `mq_message` outbox。
   - 扣减 `seckill_stock_bucket.saleable_quantity` 并递增分桶 `version`。
   - 写 `seckill_stock_change_log(DEDUCT)`。
   - 保存或保持 `seckill_result(PROCESSING)`。
7. 如果分桶扣减后库存为 `0`，触发下线检查。
8. 刷新缓存：总库存缓存可以延迟，分桶存活列表要尽快剔除空桶。
9. 返回 `ACCEPTED`。

失败处理：

- 选中桶库存不足时，优先回源最新存活桶列表重试一次。
- 如果仍无可用桶，返回库存不足。
- 库存不足不能写出 `DEDUCTED` 快照。
- outbox 落库失败时事务回滚，不返回 `ACCEPTED`。

### 分桶下线

分桶下线的目标是减少无效扣减。

触发条件：

- 扣减 SQL 返回后的库存值为 `0`。
- 或扣减失败后主库确认该桶库存不足。

流程：

1. 查询分桶真实库存。
2. 如果 `saleable_quantity <= 0`，把分桶状态更新为 `EMPTY`。
3. 从存活桶列表移除该 `bucket_no`。
4. 删除或更新缓存中的存活桶列表。

下线操作允许重复执行，必须幂等。

### 订单结果确认

订单成功时：

1. 消费结果消息。
2. 把快照从 `DEDUCTED` 改为 `CONFIRMED`。
3. 保存 `seckill_result(SUCCESS)`。
4. 标记结果消息已消费。

订单失败时：

1. 消费结果消息。
2. 查询快照中的 `bucket_id`。
3. 把快照从 `DEDUCTED` 改为 `RELEASED`。
4. 回补原分桶 `saleable_quantity` 并递增版本。
5. 写 `seckill_stock_change_log(RELEASE)`。
6. 如果原分桶从空变为有库存，恢复 `ACTIVE` 并加入存活桶列表。
7. 保存 `seckill_result(FAILED)`。
8. 标记结果消息已消费。

该流程延续当前 `confirmDeduction` 和 `releaseDeduction` 的语义，只是库存目标从单行 `seckill_sku` 变为快照中的分桶。

### 异步总库存汇总

中心桶不能在每次 C 端扣减时同步更新，否则会变成新热点。

建议新增 `SeckillStockAggregationJob`：

1. 扫描 `seckill_stock_change_log(status=NEW)`。
2. 按 `activity_id + sku_id` 聚合增量。
3. 更新中心桶 `saleable_quantity` 或统计字段。
4. 把流水标记为 `AGGREGATED`。
5. 刷新总库存缓存。

一致性口径：

- 分桶库存之和是精确事实。
- 中心桶总库存是异步汇总结果，允许短暂延迟。
- C 端展示可读中心桶缓存；强校验或对账读取分桶聚合。

### 库存调拨

调拨用于处理分桶库存不均，不是每次扣减失败都必须执行。

触发条件：

- 选中桶不足，但其他桶仍有库存。
- 存活桶数量低于阈值。
- 某些桶库存碎片过低，影响成功率。

首版建议只做受控同步调拨或后台调拨，不在高频扣减线程内做复杂调拨。

调拨流程：

1. 以目标桶获取分布式锁，例如 `seckill:bucket:transfer:{activityId}:{skuId}:{targetBucketNo}`。
2. 选择来源桶，优先选择库存较多的 ACTIVE 桶。
3. 在事务内扣来源桶、加目标桶，写两条变更流水。
4. 更新来源桶和目标桶状态。
5. 更新存活桶列表。
6. 释放锁。

失败策略：

- 未拿到锁直接跳过调拨，不阻塞高频请求。
- 来源桶不足时换桶一次或放弃。
- 调拨失败不能影响已经成功的订单扣减。

### 碎片整理

碎片整理是低优先级后台能力。只有在低库存后期、失败率升高且调拨不足以解决时启用。

流程：

1. 周期性扫描低库存分桶。
2. 选择多个低库存桶，把零散库存合并到一个目标桶。
3. 使用调拨同一套锁和流水机制。
4. 更新存活桶列表。

首个实现版本可以只保留设计和指标，不落地碎片整理任务。

## 缓存设计

### Key 规划

建议新增：

| Key | 含义 |
| --- | --- |
| `seckill:stock-total:{activityId}:{skuId}` | 总库存展示缓存 |
| `seckill:stock-buckets:{activityId}:{skuId}` | 存活桶列表缓存 |
| `seckill:stock-bucket:{activityId}:{skuId}:{bucketNo}` | 单桶库存版本缓存，可选 |

### 写入原则

- 数据库仍是事实源。
- 总库存缓存由异步汇总刷新。
- 存活桶列表缓存由初始化、下线、回补和调拨刷新。
- 版本号仍用于防止旧值覆盖新值。
- 缓存异常时回源数据库或走保守失败，不得产生超卖。

## 分片键设计

单库演示版本可以先使用 `shard_key` 字段但不实际路由。生产版本必须让同一 SKU 的不同桶尽量分散到不同逻辑分片或物理实例。

设计原则：

- `bucket_no` 不能直接等同于物理分片。
- `shard_key` 要可计算、可复现、可审计。
- 同一活动 SKU 的多个分桶应尽量均匀分布。
- 扩容时尽量保持旧桶映射稳定，避免重建全部库存。

生产模式可以参考原文的 int64 MOD 构造法：为每个桶计算一个目标逻辑分片，再构造满足路由规则的 `shard_key`。当前项目文档中只保留接口和字段，真实路由实现作为后续任务。

## 观测与验收

### 核心指标

| 指标 | 含义 |
| --- | --- |
| `seckill.bucket.deduct.total` | 分桶扣减事务总耗时 |
| `seckill.bucket.deduct.update` | 分桶库存 UPDATE 耗时 |
| `seckill.bucket.route.total` | 选择桶耗时 |
| `seckill.bucket.route.empty` | 无存活桶次数 |
| `seckill.bucket.route.retry` | 换桶重试次数 |
| `seckill.bucket.offline.total` | 分桶下线次数 |
| `seckill.bucket.release.total` | 分桶回补耗时 |
| `seckill.bucket.transfer.total` | 调拨次数和耗时 |
| `seckill.bucket.aggregate.lag` | 总库存异步汇总延迟 |
| `seckill.bucket.active.count` | 存活桶数量 |

### 账本验收

每轮压测后必须验证：

- `CONFIRMED` 快照数等于秒杀订单数。
- `RELEASED` 快照对应订单失败数。
- 所有分桶库存之和等于初始库存减确认订单数。
- 中心桶异步汇总最终等于分桶库存之和。
- 不存在 `DEDUCTED` 长期残留，除非对应 outbox 仍在补偿窗口内。
- 同一活动同一用户仍只成功一次。
- 回补必须回到快照记录的原分桶。

### 性能验收

单库演示版本验收：

- 分桶扣减成功分布大致均匀。
- 多分桶版本相比单行版本减少单行排队，但不宣称生产 QPS。
- 空桶下线后无效扣减明显下降。
- 回补后分桶可重新参与扣减。

生产分片版本验收：

- 每个物理分片的扣减流量接近均匀。
- 分桶数增加后吞吐接近线性提升，直到入口、应用线程或 MQ 变成新瓶颈。
- 中心桶更新不随 C 端 QPS 线性增长。

## 分阶段落地建议

### Stage 3A：单库分桶闭环

完成数据模型、初始化、路由、分桶扣减、快照回补和基础验收。该阶段只证明模型正确。

### Stage 3B：异步汇总和缓存闭环

补齐中心桶异步汇总、总库存缓存、存活桶列表缓存和修复任务。该阶段证明最终一致性。

### Stage 3C：下线、上线和调拨

补齐空桶下线、回补上线、受控调拨和调拨指标。该阶段提升低库存成功率。

### Stage 3D：生产分片适配

接入真实分片键计算和多实例压测。该阶段才可以讨论接近原文的水平扩展能力。

## 风险与取舍

- 分桶会提升吞吐上限，但会引入更多最终一致性状态，需要更强的对账和修复能力。
- 中心桶异步汇总牺牲实时展示精度，换取扣减链路不产生新热点。
- 调拨能降低低库存失败率，但调拨本身如果设计过重，会在低库存阶段制造新热点。
- 当前项目单库演示无法证明 8w QPS，只能证明阶段三模型和正确性闭环。
- 快照字段扩展后，旧数据需要兼容读取；没有 `bucket_id` 的旧快照仍按旧单行库存处理。
